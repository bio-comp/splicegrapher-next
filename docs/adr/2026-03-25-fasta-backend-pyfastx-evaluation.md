# ADR: FASTA Backend Candidate Evaluation (pyfastx)

- Date: 2026-03-25
- Status: Accepted
- Issue: `#53`

## Context

SGN currently uses a pyfaidx-backed path for FASTA files and supports:

- FASTA path input (`str` / `PathLike`)
- open file handle input (`IO[str]` / `IO[bytes]`)
- gzip input via existing stream helpers
- long-header lookup fidelity

Issue `#53` requested evaluation of `pyfastx` as a backend candidate and a decision:
adopt default, adopt optional, or reject.

## Evidence

Probe artifact:

- `docs/testing/2026-03-25-fasta-backend-probe.json`
- generated by:
  - `uv run --with pyfastx python benchmarks/fasta_backend_probe.py --records 4000 --seq-len 300 --iterations 5 --random-access-queries 600 --output docs/testing/2026-03-25-fasta-backend-probe.json`

Observed results from that run:

- parity:
  - `pyfaidx_matches_sg = true`
  - `pyfastx_matches_sg = true`
- file-handle contract:
  - `sg_supports_file_handles = true`
  - `pyfastx_supports_file_handles = false`
- iteration speed (mean seconds):
  - `sg = 0.01184`
  - `pyfaidx = 0.01112`
  - `pyfastx = 0.00426`
- random access speed (mean seconds, 600 lookups):
  - `sg = 5.33590`
  - `pyfaidx = 0.00967`
  - `pyfastx = 0.00326`
- index build probe:
  - `pyfaidx`: `probe.fa.fai` (108,585 bytes), 0.00857 s
  - `pyfastx`: `probe.fa.fxi` (339,968 bytes), 0.00386 s

## Decision

Reject `pyfastx` as the default SGN FASTA backend for now.

Keep pyfaidx as the default path backend in runtime code.

## Rationale

1. Contract mismatch:
- SGN FASTA currently supports open file handles; pyfastx does not.
- Default backend swap would be a behavior change for existing callers.

2. Sidecar/index implications:
- pyfastx introduces `.fxi` sidecars with materially different size/shape expectations
  than current `.fai` behavior.
- this would alter filesystem footprint and operational assumptions in downstream workflows.

3. Performance alone is insufficient for default replacement:
- pyfastx is faster in this probe, but default adoption must preserve contract compatibility
  first during SGN cleanup freeze.

## Consequences

Positive:

- SGN runtime behavior remains stable for existing FASTA call patterns.
- Issue `#53` closes with explicit evidence and recommendation.

Negative:

- SGN does not gain pyfastx performance by default in this tranche.

Follow-up:

- optional backend exploration can be revisited in a dedicated issue if SGN decides to
  introduce explicit backend selection and handle contract migration.
